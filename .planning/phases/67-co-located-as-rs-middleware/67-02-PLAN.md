<pkce_rejection_note>
The S256-only ENFORCEMENT (ROADMAP SC#1 / design doc lines 292-309) must actively reject:
  (a) code_challenge_method=plain      -> 400 invalid_request, error_description="PKCE plain method not supported"
  (b) code_challenge present, method absent -> 400 invalid_request
  (c) code_challenge absent entirely   -> 400 invalid_request
The SDK's AuthorizationParams does not surface code_challenge_method. The provider's
authorize() receives params with code_challenge already parsed. To enforce (a)/(b)/(c) as a
CONTRACT TEST, the executor MUST inspect the raw /authorize query params. Approach: have
the provider.authorize() reject when params.code_challenge is falsy (covers (c)); AND add a
thin pre-check helper `reject_non_s256_pkce(query_params: Mapping[str,str]) -> None` in
provider.py that raises an SDK AuthorizeError / returns the 400 error redirect for plain or
method-absent (covers (a)/(b)). The contract test calls this helper directly with the three
bad inputs and asserts the 400 invalid_request + exact error_description string. Wiring the
helper into the live /authorize route happens in Plan 04 when create_auth_routes is mounted;
this plan delivers + unit-tests the helper and the provider rejection.
</pkce_rejection_note>
