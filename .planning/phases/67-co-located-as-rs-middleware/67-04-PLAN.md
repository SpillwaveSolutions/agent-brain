<token_validation_order>
Design doc §"Token Validation on /mcp" lines 311-324 — checks #1-5 in Phase 67 (#6 scope is Phase 68):
  1. Bearer present -> else 401
  2. RS256 signature valid (verified against the in-memory public key / local JWKS)
  3. exp not expired + nbf honored, leeway ~30s
  4. iss == AGENT_BRAIN_OAUTH_ISSUER (or co-located AS base URL)
  5. aud == AGENT_BRAIN_OAUTH_RESOURCE  (RFC 8707) -> mismatch rejected
Failure #1-5 -> 401 + WWW-Authenticate: Bearer resource_metadata="...".
Do NOT implement scope check (#6) here — but keep request.state.auth claims reachable for Phase 68.
</token_validation_order>

<pkce_live_wiring_note>
ROADMAP SC#1 + CONTEXT §"PKCE S256-only rejection gate" require the LIVE /authorize route to
actively REJECT non-S256 PKCE with HTTP 400 invalid_request — as a contract test, not just
OASM advertisement. The SDK `AuthorizationParams` model does NOT carry code_challenge_method,
so the provider's authorize() can only detect case (c) absent-challenge — NOT (a) plain or
(b) challenge-present-method-absent. Those require inspecting the RAW HTTP query params before
the SDK parses them — i.e. in http.py at the route layer. Plan 02 delivered
`reject_non_s256_pkce(query_params)` and explicitly DEFERRED live-route wiring to this plan.
This plan adds a thin auth-exempt /authorize front-handler that runs reject_non_s256_pkce
BEFORE delegating to the SDK-generated authorize handler.
</pkce_live_wiring_note>
