<objective>
Make `/health/status` graph counts true. Today the graph `entity_count` /
`relationship_count` come from the in-memory `self._entity_count` /
`self._relationship_count` bookkeeping in graph_store.py, which DRIFTS after a
job-timeout rollback — producing the #184 `0 / 100` vs real `5677 / 4366`
discrepancy. Replace that source with a LIVE kuzu `COUNT(*)` evaluated at query
time, wrapped in a short (~5s) TTL cache so health-poll storms don't issue a graph
query per request. Only run when graph is enabled AND `store_type == kuzu`. If the
live COUNT itself fails (kuzu unreachable), report last-known counts with a
degraded/stale marker — NEVER `0` (which reads as "empty graph" and hides an outage).

Purpose: health numbers match reality (Phase 64 Success Criterion 3).
Output: live_counts() with TTL cache + degraded fallback, wired into get_status, + tests.
</objective>
