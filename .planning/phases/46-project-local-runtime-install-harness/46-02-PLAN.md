Extend `e2e-cli/lib/runtime_parity.sh` with shared helpers that implement the approved three-stage verification contract and dual failure output:

1. Add a helper that writes `${runtime_root}/logs/failure.log` with a plain-text timestamped summary.
2. Add a helper that returns structured JSON objects with exactly these keys: `runtime`, `status`, `error_type`, `details`, `remediation`, and `workspace`.
3. Add a shared `runtime_verify_install` helper that takes runtime name, workspace root, expected target path, and an optional dry-run command. It must execute these gates in order:
   - structure check: verify the runtime-specific install target directory exists and key generated files exist for that runtime
   - targeted file/JSON validation: parse the install-agent JSON and verify `target_dir` matches the expected project-local target
   - dry probe: run a runtime-specific probe callback or command that must return valid JSON; if parsing fails, emit `error_type: malformed_json`

Implement explicit failure types at minimum for:
- `missing_cli`
- `install_verification_failed`
- `malformed_json`
- `forbidden_global_path`
- `global_path_mutated`

For each failure type, populate `remediation` with concrete operator guidance. Examples:
- missing CLI -> install the runtime binary or place a fake binary in PATH for tests
- malformed JSON -> inspect the runtime command and preserve raw output in logs
- forbidden/global mutation -> use `--project --path <repo-owned-dir>` and inspect the offending target

Keep the helper shell-only and deterministic. Do not introduce Python or jq dependencies for JSON validation if the repo’s shell test pattern can avoid them.
