       ```markdown
       # Agent Brain Query Benchmarks

       ## Overview

       This document records baseline query latency measurements for Agent Brain.
       Benchmarks use the reproducible runner at `scripts/query_benchmark.py` with a
       fixed query set (`scripts/benchmark_queries.json`).

       ## How to Run

       ### Quick Start

       ```bash
       # Start server and ensure docs/ is indexed, then:
       python scripts/query_benchmark.py

       # With automatic corpus setup:
       python scripts/query_benchmark.py --prepare-docs-corpus

       # JSON output for CI:
       python scripts/query_benchmark.py --json
       ```

       ### Options

       | Flag | Default | Description |
       |------|---------|-------------|
       | `--server-url` | `http://127.0.0.1:8000` | Server URL |
       | `--iterations` | `20` | Measured queries per mode |
       | `--warmups` | `3` | Warm-up queries per mode (discarded) |
       | `--modes` | all | Comma-separated modes to benchmark |
       | `--json` | off | Output JSON instead of Rich table |
       | `--prepare-docs-corpus` | off | Reset index and re-index `docs/` before benchmarking |
       | `--queries-file` | `scripts/benchmark_queries.json` | Path to query set |

       ## Methodology

       - **Transport**: HTTP POST to `/query` endpoint (measures user-visible latency)
       - **Timing**: Dual -- client-observed end-to-end latency and server-reported `query_time_ms`
       - **Warm-up**: 3 queries per mode discarded before measurement
       - **Iterations**: 20 measured queries per mode
       - **Corpus**: Repository `docs/` folder
       - **Pool settings**: Default `pool_timeout: 30` (tunable via config.yaml)
       - **Unsupported modes**: Reported explicitly (e.g., `graph` on PostgreSQL)

       ## Baseline Results

       > **Note:** Run `python scripts/query_benchmark.py --prepare-docs-corpus` against
       > your local setup to generate your own baseline. The numbers below are reference
       > values from the initial benchmark run.

       ### Run Metadata

       | Field | Value |
       |-------|-------|
       | Date | _(to be filled after baseline run)_ |
       | OS | _(to be filled)_ |
       | Python | _(to be filled)_ |
       | Backend | ChromaDB |
       | GraphRAG | _(enabled/disabled)_ |
       | Iterations | 20 |
       | Warm-ups | 3 |
       | Corpus | `docs/` |
       | Chunk Count | _(to be filled)_ |

       ### Client-Observed Latency (ms)

       | Mode | p50 | p95 | p99 | Mean | QPS |
       |------|-----|-----|-----|------|-----|
       | vector | - | - | - | - | - |
       | bm25 | - | - | - | - | - |
       | hybrid | - | - | - | - | - |
       | graph | - | - | - | - | - |
       | multi | - | - | - | - | - |

       ### Server-Reported query_time_ms

       | Mode | p50 | p95 | p99 | Mean |
       |------|-----|-----|-----|------|
       | vector | - | - | - | - |
       | bm25 | - | - | - | - |
       | hybrid | - | - | - | - |
       | graph | - | - | - | - |
       | multi | - | - | - | - |

       > Replace `-` with actual numbers after running the benchmark.
       > Unsupported modes will show "unsupported" in the Status column.

       ## Interpreting Results

       - **p50** (median): Typical query latency
       - **p95/p99**: Tail latency -- important for user experience
       - **QPS**: Queries per second throughput
       - **Client vs Server**: Large gap indicates network/serialization overhead
       - **graph mode**: Only available on ChromaDB with GraphRAG enabled
       - **multi mode**: Combines available retrieval modes; on PostgreSQL, graph contribution is skipped

       ## Regression Detection

       Compare new runs against the baseline table. A >20% increase in p95 latency
       for any mode warrants investigation.
