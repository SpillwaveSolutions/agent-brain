<spec_verification_note>
The live MCP Authorization spec (https://modelcontextprotocol.io/specification/draft/basic/authorization)
was fetched during planning on 2026-06-14. Confirmed normative levels the doc MUST cite:
- MCP servers MUST implement RFC 9728 Protected Resource Metadata; clients MUST use it for AS discovery.
- AS + clients SHOULD support Client ID Metadata Documents (CIMD, draft-ietf-oauth-client-id-metadata-document-00).
- AS + clients MAY support DCR (RFC 7591) — explicitly "deprecated and retained for backwards compatibility".
- RFC 8707 `resource` param MUST be in BOTH authorization and token requests; RS MUST validate the token was issued for it as audience (aud).
- Canonical URI examples: `https://mcp.example.com/mcp`; SHOULD omit trailing slash; MUST have scheme; MUST NOT contain fragment.
- Insufficient scope at runtime → HTTP 403 + `WWW-Authenticate: Bearer error="insufficient_scope", scope="...", resource_metadata="..."`. 401 = invalid/missing token; 403 = scope.
- RFC 9207 `iss` validation: AS SHOULD include `iss`; clients MUST validate against recorded issuer. (Present in live draft; a future revision is expected to upgrade SHOULD->MUST.)
- DPoP is NOT in the core authorization spec — it lives in the optional/additive ext-auth extensions repo. Deferring it therefore violates no core-spec MUST.
- The live draft page does NOT surface an explicit "2026-07-28 RC / stateless / no-initialize" change to the authorization flow as of the 2026-06-14 fetch. The doc must record this finding (the staleness risk is acknowledged but not yet realized in the authorization spec).

The executor MUST re-verify the live spec at authoring time (context7 `/modelcontextprotocol/modelcontextprotocol` or WebFetch the draft authorization URL) and record in the doc what the live spec shows on the authoring date, including whether the 2026-07-28 RC has landed.
</spec_verification_note>
