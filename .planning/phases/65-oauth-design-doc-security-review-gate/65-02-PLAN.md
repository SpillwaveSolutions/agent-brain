and a recorded human approval.

<review_targets>
The four converged risks the review MUST adversarially probe (from research SUMMARY
"Critical Pitfalls" and 65-CONTEXT threat model):
  1. Confused-deputy / token-passthrough — does the doc guarantee the client OAuth
     token terminates at the MCP boundary and the MCP->REST leg keeps AGENT_BRAIN_API_KEY?
  2. aud-claim omission — does the doc require RS-side aud == canonical URI validation
     on every inbound token, and AS-side aud binding via the resource param (RFC 8707)?
  3. Well-known-behind-auth deadlock — does the doc mandate that /.well-known/*,
     /authorize, /token, jwks.json mount OUTSIDE RequireAuthMiddleware (200 with no token)?
  4. Per-tool scope escalation — does the doc enforce per-tool scope (not just token
     presence), returning 403 (not 401) on insufficient scope?
Also probe: CIMD SSRF allowlist completeness; FileTokenStorage chmod 0o600 / Pattern A
token reuse; PKCE S256-only enforcement (reject plain/absent); the in-memory token-store
restart trade-off; the 2026-07-28 RC staleness acknowledgement.
</review_targets>
</context>
