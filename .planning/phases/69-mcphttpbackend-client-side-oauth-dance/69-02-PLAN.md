<objective>
Create the two browser/loopback callables the SDK `OAuthClientProvider` requires:

1. A `redirect_handler: Callable[[str], Awaitable[None]]` that opens the system
   browser to the authorization URL AND prints the URL to stderr ("Open this URL
   to authorize: ...") as a headless fallback (decision C).

2. A `callback_handler: Callable[[], Awaitable[tuple[str, str | None]]]` backed by
   an ephemeral localhost HTTP server bound to an OS-assigned port. It receives the
   OAuth provider's redirect (`GET /callback?code=...&state=...`), captures `code`
   and `state`, returns a friendly "authentication complete â€” you may close this
   tab" HTML page, then shuts down and returns `(code, state)`.

Because the SDK registers the client with `redirect_uris` BEFORE the dance, the
loopback port must be chosen up front. This plan therefore exposes the chosen
`redirect_uri` (`http://127.0.0.1:<port>/callback`) so the Wave-2 factory can put
it in the `OAuthClientMetadata.redirect_uris`.

Purpose: These are the only bespoke UX pieces of the dance â€” everything else is
SDK-driven. They satisfy SC#1's "first invocation opens a browser" bar.

Output: `agent_brain_mcp/oauth/oauth_handlers.py` (net-new) + unit tests that
simulate the callback GET without a real browser.
</objective>
